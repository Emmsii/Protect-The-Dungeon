How the database shit works.

strike through = done.
  • Player registers account on website.
  • Player can now login to app. (Even if user doesn't exist/wrong pass, player can still log in atm. App can tell if user exists, just cant GET user data yet)
  • Password stored with hash.
  • When player enters username and password in application, sends GET url with user and pass. (Make this safer with hashing?)
  • If user and pass are correct, get some data from the database.
  • If not, app commits seppuku. (I'll think of something to happen if player is offline/bad login/website down. For the moment just kill app.)
  • Player comes to character select screen. Max three characters.
  • Get character(s) info: id, name, stats, dugneon owned id.
  • If no characters exist, player can create one. Character details are POSTed to database.
  • Get dungeon info based of selected character id (if character has dungeon built): id, room_string, obj_string.
  • When saving dungeon, app sends POST url with dungeon info (see above).
  • When player loads world map (see below about world map), app sends GET url to return list of dungeons.
  • When player chooses dungeon to attack, app sends GET url to receive dungeon data.
  • POST url is sent to say that dungeon is currently under attack. Dungeon cannot be attacked by other players.
  • If player wins/looses/out of time, app sends POST url to update dungeon attack status.
  • After every attack, no matter what the outcome, app sends POST url with updated character stats.
    World Map
  • Each dungeon has random x, y coords.
  • When world map is requested, dungeon icons are placed in an area based off x, y.
  • When new dungeon is created its given unused x, y, coords.
